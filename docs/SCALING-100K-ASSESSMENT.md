# Оцінка: чи витримає сайт 100k онлайн

## Висновок

**100k одночасно активних користувачів (одночасний онлайн)** — **ні**, поточної архітектури недостатньо. Потрібні зміни в кешуванні, лічильнику онлайн і обмеженнях хостингу.

**100k унікальних відвідувачів на день** при піку **до ~3–5k одночасних** — **так**, реально, якщо кеш новин і KV працюють стабільно.

---

## Що відбувається зараз на кожного користувача

| Дія | Частота | API |
|-----|--------|-----|
| Завантаження сторінки | 1 раз | GET /api/news, GET /api/online, POST /api/online |
| Оновлення новин | кожні 3 хв | GET /api/news |
| Оновлення «онлайн» | кожні 25 с | GET /api/online |
| Пульс «я тут» | кожні 50 с | POST /api/online |

При **100k одночасних** у першу хвилину це приблизно:
- **~100k** запитів GET /api/news
- **~100k** GET /api/online + **~100k** POST /api/online
- далі ще **~240k** GET /api/online і **~120k** POST /api/online на хвилину

---

## Вузькі місця

### 1. **Vercel (serverless)**

- Ліміти **одночасних виконань** (concurrent executions) на план (наприклад, сотні–тисяча).
- При 100k одночасних запитах черга буде величезна, частина запитів — таймаут або 503.
- **Висновок:** без edge-кешу або окремого кеш-шару перед функціями 100k одночасних не витримати.

### 2. **GET /api/news**

- Якщо **є кеш по категорії** (KV): один великий JSON віддається з KV, але далі для **кожного з ~30 новин** робиться окремий **kv.get** (мітка «Важливе/Інформативне/Корисне») і при потребі **kv.set**.
- Тобто **~30–60 операцій KV на один запит**. 100k запитів ≈ **3–6 млн операцій KV** за короткий час — це перевищує типовий throughput Vercel KV / Upstash.
- Якщо **немає кешу** (cold): додатково **5–6 зовнішніх HTTP-запитів** (RSS) на кожен запит; зовнішні сайти можуть обмежувати або блокувати.
- **Висновок:** поточна схема (окремі get/set на кожну новину) при 100k онлайну не масштабується. Потрібно кешувати вже **готовий список із мітками** (наприклад, один ключ на категорію з повним JSON).

### 3. **GET /api/online**

- Лічильник будується через **SCAN по префіксу** `ln_online:*` (перебір ключів).
- При **100k онлайн** це **100k ключів**; кожен GET викликає такий перебір. У коді використовується `scanIterator` з `count: 200`, тобто фактично рахується не більше 200 ключів — **число «онлайн» обрізається** і не відображає справжні 100k.
- Навантаження на KV: **сотні тисяч SCAN-операцій на хвилину** — дуже важко для Redis/KV.
- **Висновок:** при 100k онлайну поточний підхід (scan по ключах) не підходить. Потрібен окремий лічильник (один ключ + INCR/DECR або подібне) або зовнішній сервіс для метрик.

### 4. **POST /api/online (пульс)**

- На кожен запит: rate limit (get+set) + 2× set (онлайн + день). Це **кілька операцій KV на користувача** кожні 50 с.
- При 100k користувачах — **~120k POST на хвилину**, тобто **~360k+ операцій KV/хв** тільки на пульси. Це на межі або вище типових лімітів.
- **Висновок:** можливо, але разом з іншими запитами (новини, GET online) KV стане вузьким місцем.

### 5. **База (Neon) та інші API**

- `/api/news` викликає `getNewsForPeriod` — додаткове навантаження на БД при кожному запиті новин.
- Digest, send-news тощо при 100k онлайну теж дадуть піки. Їх частоту варто обмежувати або виносити в черги.

## Що змінити, щоб наблизитися до 100k онлайну

1. **Новини — один кеш на категорію вже з мітками**  
   Зберігати в KV не лише список новин, а **готовий список із полем `label`** і віддавати його одним get. Мітки рахувати при оновленні кешу (наприклад, по крону або при першому запиті після TTL), а не на кожен запит. Це зменшить кількість KV-операцій з мільйонів до одиниць на запит.

2. **Лічильник онлайн без SCAN**  
   Використовувати один ключ типу `ln_online_count` і змінювати його через INCR при POST (пульс) і зменшувати через TTL або окремий механізм (наприклад, приблизний підрахунок по TTL 120 с і оцінка по кількості унікальних пульсів за останні 2 хв). Або винести підрахунок онлайн у зовнішній сервіс (наприклад, аналітика/метрики).

3. **Кешування відповідей API на edge**  
   Вмикати для GET /api/news і, за можливості, для GET /api/online HTTP-кеш (Cache-Control, CDN/Vercel Edge) з TTL 1–3 хв, щоб більшість запитів не доходили до serverless і KV.

4. **Рідші запити з клієнта**  
   Збільшити інтервал оновлення новин (наприклад, 5–10 хв замість 3) і інтервал GET /api/online (наприклад, 60 с замість 25). Це значно знизить RPS і навантаження на KV.

5. **План і ліміти Vercel/KV**  
   Перевірити максимальну кількість одночасних виконань функцій і throughput KV на поточному плані. Для дуже високого навантаження може знадобитися виділений Redis або інший кеш поза Vercel.

## Орієнтовні цифри

| Сценарій | Одночасних | Реалістично? |
|----------|------------|--------------|
| 100k одночасно на сайті | 100 000 | Ні (без змін вище) |
| 5–10k одночасно, 100k+ унікальних/день | 5 000–10 000 | Можливо, з сильним кешем новин і оптимізацією online |
| 1–3k одночасно | 1 000–3 000 | Так, при стабільному KV і кеші |

Підсумок: **сайт не розрахований на 100k одночасних онлайну без змін**. З кешем повного списку новин із мітками, переробленим лічильником онлайн і edge-кешем — можна суттєво підняти стельу (орієнтовно до десятків тисяч одночасних, залежно від плану та KV).
